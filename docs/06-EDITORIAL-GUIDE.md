# Editorial Guide

## Voice

Use clear, practical English. Avoid hype, vague claims, and generic tool marketing.

## Structure

Most pages should include:

- A short definition or verdict
- Best-fit use cases
- Non-fit cases or limitations
- Practical next steps
- Internal links to related stacks, kits, guides, or tools

## Comparison Pages

Comparison pages should help a reader choose. They must include criteria that matter for local-first workflows:

- Data ownership
- Export and migration
- Offline behavior
- Git support
- Publishing fit
- Collaboration fit

## Guide Pages

Guides should focus on one action. Include prerequisites, steps, commands when useful, and a clear acceptance check.

## Style Rules

- Use English only.
- Keep paragraphs short.
- Prefer concrete workflow language over abstract product language.
- Do not copy long official product descriptions.
- Link to official sites instead of mirroring documentation.
